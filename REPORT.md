¿Cuál es el bug raíz y en qué archivo/función está?
El error fundamental se encuentra en la implementación del algoritmo de cifrado en el subsistema de criptografía del kernel, específicamente en el archivo crypto/algif_aead.c dentro de la función aead_recvmsg_sync. El problema reside en una gestión incorrecta de los índices de memoria al manipular datos asociados (AD) y datos cifrados. La función no valida correctamente los límites del buffer de destino cuando se concatenan los resultados, permitiendo una escritura fuera de límites
¿Por qué el write a dst[assoclen + cryptlen] es peligroso?
La escritura en la posición dst[assoclen + cryptlen] es extremadamente peligrosa porque asume que el buffer de destino siempre tiene espacio suficiente para la suma de la longitud de los datos asociados y la longitud del texto cifrado. Al no existir una verificación de seguridad, un atacante puede manipular estos valores para escribir datos arbitrarios en una región de memoria adyacente. En el contexto del kernel, esto significa que podemos sobrescribir estructuras de datos críticas, como punteros de función o punteros a objetos del sistema, lo que abre la puerta a la ejecución de código o escalada de privilegios.
¿Por qué el exploit es "stealthy" (no modifica el archivo en disco)?
Este exploit se considera "sigiloso" porque no necesita modificar archivos físicos en el almacenamiento persistente, disco. En lugar de alterar el binario de un ejecutable como /bin/sh directamente en el disco, el exploit manipula la memoria del sistema para engañar al kernel. Al atacar la memoria dinámica, el archivo original permanece intacto, lo que hace que las herramientas de verificación de integridad de archivos tradicionales no detecten ninguna anomalía.
Conecta esto con lo que vimos en clase: page cache, chmod, setuid, inodos
El éxito de este ataque depende de entender cómo interactúan los componentes del kernel:
Page Cache: El exploit aprovecha cómo el kernel mantiene páginas de memoria en caché para mejorar el rendimiento de archivos. Al sobrescribir una página en la memoria RAM que pertenece a un binario con permisos de setuid, podemos alterar su comportamiento mientras se ejecuta.
Inodos y chmod: Aunque el inodo del archivo en disco indica permisos específicos (como un chmod 4755 para un binario setuid), nuestra manipulación ocurre a nivel de memoria volátil. Engañamos al sistema para que ejecute nuestra carga útil con los privilegios del propietario del archivo (root), utilizando la estructura de permisos existente a nuestro favor.
¿Qué aprendiste sobre cómo múltiples cambios "razonables" pueden crear un bug grave?
Este caso es una lección magistral sobre la seguridad en el desarrollo de software. Aprendí que un bug grave no siempre nace de un error catastrófico evidente, sino de la suma de múltiples cambios que, por sí solos, parecen lógicos y razonables. Optimizar una función para que sea más rápida o simplificar una suma de longitudes parece inofensivo, pero en sistemas complejos como el kernel de Linux, cualquier suposición no validada se convierte en una superficie de ataque. La seguridad debe ser integral; un solo descuido en la validación de un índice puede invalidar miles de líneas de código seguro.
